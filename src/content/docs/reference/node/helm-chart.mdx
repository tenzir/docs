---
title: Helm chart
---

The Tenzir Helm chart deploys one or more `tenzir-node` instances on a
Kubernetes cluster. It deploys nodes only — it does not deploy the Tenzir
Platform. Nodes installed through the chart connect to a Platform you have
already provisioned, either cloud-hosted at `app.tenzir.com` or self-hosted
through the Sovereign Edition.

The chart is distributed as an OCI artifact at
`oci://ghcr.io/tenzir/charts/tenzir-node`. The source and the full values
reference live in
[`tenzir/helm-charts`](https://github.com/tenzir/helm-charts/tree/main/charts/tenzir-node).
For step-by-step installation, see
<Guide>node-setup/deploy-a-node#kubernetes</Guide>.

## Resources rendered per node

Each entry in the chart's `nodes` list renders an independent set of
Kubernetes resources for one Tenzir Node:

- A `StatefulSet` with one replica running `tenzir-node`.
- A ClusterIP `Service` and a headless `Service` on port `5158`.
- A `ConfigMap` holding the rendered `tenzir.yaml`.
- A `Secret` containing the node's `TENZIR_TOKEN`, unless you reference an
  existing one with `nodes[].token.existingSecret`.
- A `PersistentVolumeClaim` for `/var/lib/tenzir` when
  `persistence.enabled: true`.
- A test `Pod` (a `helm test` hook) that calls `tenzir api "/ping"`.

Each entry has an independent identity. The token corresponds to one node
provisioned in your Tenzir Platform workspace, the persistent state belongs
to that node alone, and pods never share volumes.

Cluster-wide resources rendered once per release:

- A `ServiceAccount` when `serviceAccount.create: true`.
- A `NetworkPolicy` when `networkPolicy.enabled: true`.
- A `PodDisruptionBudget` when `podDisruptionBudget.minAvailable` or
  `.maxUnavailable` is set.
- One `Service` per `sharedServices[]` entry.

## Per-node configuration

The chart composes each node's `tenzir.yaml` from three layers, merged in
order:

1. Built-in chart defaults: `tenzir.endpoint: 0.0.0.0:5158`,
   `tenzir.file-verbosity: quiet`, `tenzir.console-sink: stderr`.
2. The global `tenzir.config` overlay (shared across every node in the
   release).
3. The per-node `nodes[].config` overlay (specific to one node).

A `checksum/config` annotation on each pod template, derived from the
merged YAML, ensures that `helm upgrade` only rolls the pods whose
configuration actually changed. Nodes whose merged YAML is unchanged keep
running through the upgrade.

To connect nodes to the Tenzir Platform, set the platform endpoint in the
global overlay:

```yaml
tenzir:
  config:
    tenzir:
      platform-control-endpoint: wss://ws.tenzir.app/production
```

## Expose listener ports

Every node always exposes the node-to-node port (`5158` by default) on its
own main and headless `Service`. The chart provides two patterns for
additional listener ports.

### One node owns the port

Use `nodes[].extraPorts` when a specific node listens on a specific port:

```yaml
nodes:
  - name: collector
    token:
      existingSecret: tenzir-collector-token
    extraPorts:
      - name: http
        containerPort: 8080
```

The chart opens the container port on that node's pod and adds the port to
that node's main `Service`. Set `serviceType: LoadBalancer` (or
`NodePort`) on an entry to publish the port through a dedicated `Service`
of that type instead.

### A fleet of nodes shares the port

Use `sharedServices` when several nodes accept the same traffic and you
want kube-proxy to load-balance across them:

```yaml
nodes:
  - name: ingester-a
    token: { existingSecret: tenzir-ingester-a-token }
  - name: ingester-b
    token: { existingSecret: tenzir-ingester-b-token }

sharedServices:
  - name: http-ingest
    port: 8080
    type: LoadBalancer
```

The chart opens the container port on every selected node's pod and
creates a single `Service` whose selector spans all of them. Each selected
node must run a pipeline that actually listens on the port — for example,
`accept_http` declared under `tenzir.pipelines` in `nodes[].config`.

## Restrict ingress with a NetworkPolicy

Set `networkPolicy.enabled: true` to render a `NetworkPolicy` that selects
every node pod in the release. By default the policy allows ingress from
any pod in the same namespace. Add explicit rules to grant access from
other namespaces or specific pods:

```yaml
networkPolicy:
  enabled: true
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: observability
      ports:
        - port: 5158
          protocol: TCP
```

Set `allowSameNamespace: false` to drop the default same-namespace rule
and rely entirely on the entries in `ingress`.

## Container hardening

The chart's default container `securityContext` clears most of Trivy's
container-hardening rules at HIGH severity:

- `runAsNonRoot: true` with `runAsUser: 999` and `runAsGroup: 999` to
  match the image's `tenzir` user.
- `seccompProfile: { type: RuntimeDefault }`.
- `allowPrivilegeEscalation: false`.
- `capabilities.drop: [ALL]`.
- `readOnlyRootFilesystem: true` with an `emptyDir` mounted at `/tmp` for
  writable temp files. `/var/lib/tenzir` stays writable through the PVC.

Two edge cases need attention:

- **Privileged listener ports.** When a pipeline binds a port below 1024
  (for example `accept_tcp "0.0.0.0:514"` for Syslog), the listener needs
  `CAP_NET_BIND_SERVICE` only if the cluster keeps the kernel default
  `net.ipv4.ip_unprivileged_port_start = 1024`. Most modern distributions
  (Docker Desktop, kind, k3s, and many managed offerings) ship with the
  value set to `0`, which makes every port unprivileged and removes the
  need for the capability. Check the value from a pod with
  `cat /proc/sys/net/ipv4/ip_unprivileged_port_start`. If you need the
  capability, add it back:

  ```yaml
  containerSecurityContext:
    capabilities:
      add: [NET_BIND_SERVICE]
  ```

- **Writes outside `/tmp` and `/var/lib/tenzir`.** With
  `readOnlyRootFilesystem: true`, any pipeline that calls `to_file`
  against a path outside the two writable mounts fails with a clean
  `IOError` without crashing the node. Mount additional writable paths
  through `extraVolumes` and `extraVolumeMounts` when you need them.
